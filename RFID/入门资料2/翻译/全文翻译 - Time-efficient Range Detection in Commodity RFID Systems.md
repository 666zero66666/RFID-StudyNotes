## 摘要

RFID 在我们的日常生活中正变得越来越普遍。在 RFID 标签被部署用于标识所附着的物体之后，一个自然的下一步就是与这些标签通信并收集其信息，以便实时跟踪被标记物体或者监测其周边环境。本文研究商品化 RFID 系统中一个尚未被充分探索的问题：范围检测（range detection）。该问题的目标是：高时间效率地检查是否存在某些标签，其数据位于给定上界与下界之间。

这个问题在大型 RFID 系统中尤为重要，因为它可以帮助用户快速定位目标标签（如果存在），并在风险刚刚萌芽时及时发出预警，使用户能够采取紧急措施、降低潜在损失。本文在 C1G2 标准范围内提出了两种定制协议：Selective Query 和 Range Query（RQ），用于实现范围检测。其创新点在于：不再逐个查询每个标签，而是利用 C1G2 兼容的选择能力，通过让绝大多数标签静默，快速把目标标签与其他标签区分开。最终，最优协议 RQ 只需一个查询命令即可完成范围检测。作者在商品化 RFID 系统上实现了这些协议，不需要对硬件做任何修改。大量实验表明，相比基线方案，RQ 可将时间效率提升近 30 倍。

## 1. 引言

RFID 已经广泛出现在各种应用中，包括图书馆盘点、目标跟踪、仓库管理等。在 RFID 标签被部署用来唯一标识附着对象之后，一个很自然的下一步就是与标签通信并收集感兴趣的信息。这些信息既可以是预先写入标签存储器中的静态数据，用于表示被标记商品的属性，例如保质期；也可以是在带传感能力的 RFID 系统中用于单件监测的动态感知数据，例如温度。

通过收集这些标签信息，系统既可以跟踪被标记物体的状态，也可以实时监测其周围环境。

本文研究的是商品化 RFID 系统中一个尚未被充分研究的问题：范围检测。它的目标是判断，是否存在标签，其数据位于给定的上下边界之间。论文把这些满足条件的标签称为目标标签。高时间效率在大型 RFID 系统中特别重要，因为这能帮助用户迅速定位目标标签，并在风险初现阶段就得到预警，从而采取紧急措施、降低潜在损失。

例如，在超市中，工作人员可能需要实时查询是否存在过期商品，这就需要从标签中收集销售截止日期。再例如，在冷藏食品仓库或图书馆中，每个食品或书架都可以附着带有温度传感器的 RFID 标签。如果某个标签报告的温度超过阈值，就应当提前发出预警，以防止食品变质或火灾发生，从而保护人员和财产。

一种直观的范围检测方式是：逐个查询每个标签，并检查其所持数据是否落在给定范围内。该方法虽然不会出错，但在大规模 RFID 系统中耗时很长。在某些场景下，目标标签数量远少于其他标签，对整个标签集合逐个盘点会非常浪费时间。

近年来，已经有一些工作尝试更高效地收集标签信息。例如，Chen 等人设计了多哈希方法，以避免标签碰撞并提高时间效率；Yue 等人在多读写器 RFID 系统中使用分布式 Bloom Filter 进行快速盘点；Liu 等人设计了增量轮询协议，把轮询向量长度从 96 位大幅缩减到不足 2 位，从而节省信息收集中的轮询开销。

尽管如此，这些工作都与 EPCglobal Class1 Gen2（C1G2）标准不兼容。C1G2 是全球通用的 UHF RFID 标准，而上述工作都无法直接在商品化 RFID 系统中实现。

基于此，本文深入挖掘 C1G2 的潜力，并给出两种定制化解决方案，显著提高了商品化 RFID 系统中的范围检测效率。其基本思想是：不是逐个检查每个标签，而是先让绝大多数标签静默，再尽快锁定目标标签（如果存在）。作者发现，C1G2 中规定的 `Select` 命令很有潜力帮助实现这一点。它允许读写器选择一个标签子集，使这些标签参与后续查询。

基于这一能力，作者首先提出第一种方案 Selective Query（SQ）。SQ 根据标签数据值把标签划分成若干组。与其逐个标签盘点，不如每组只查询一个标签，再用 `Select` 让同组剩余标签静默。这样，查询次数就只与组数成正比，相比逐标签查询是一个大幅下降。

尽管如此，逐组收集数据依然存在浪费。这促使作者进一步思考：能否只做一次标签查询，而不管组数多少、标签多少？答案是可以。于是作者提出第二种协议 Range Query（RQ）。RQ 先通过若干个 `Select` 命令把所有目标标签（如果存在）与其他标签分离开，然后只对被选中的标签子集做一次盘点；只要有任何回复，就说明检测到了目标标签。这里的关键挑战在于：在事先不知道标签 ID 的情况下，如何把目标标签从整个标签集合中剥离出来。作者提出了两种技术：

- substring masking（子串掩码）
- mask combination（掩码组合）

二者联合起来，可把 `Select` 的数量压到非常少。最终，RQ 通常只需要几个 `Select`（很多情况下只需两个）加一次标签盘点，就能完成目标检测，而且与标签总数无关。

本文的主要贡献有三点：

1. 研究了商品化 RFID 系统中的范围检测问题，这一问题有助于快速定位目标标签并在风险早期降低潜在损失。
2. 深入利用 C1G2 兼容命令，提出了两种定制化方案。通过联合使用子串掩码和掩码组合，系统可以在与标签总数无关的情况下，仅用一次 inventory 来检测目标标签。
3. 在一个包含 1000 个标签的商品化 RFID 系统中实现了所提协议。大量实验表明，最优协议 RQ 可以把检测时间从 `9.6s` 急剧降低到 `0.33s`，相比基线方案提升接近 30 倍。

后续结构如下：第 2 节形式化定义范围检测问题；第 3 节给出直观解法 Exclusive Collection；第 4 节介绍 Selective Query；第 5 节详细给出 Range Query；第 6 节进行性能评估；第 7 节讨论相关工作；第 8 节总结全文。

## 2. 问题形式化

考虑一个由读写器和若干标签构成的 RFID 系统。每个标签有唯一的标签 ID，用于唯一标识相关对象。通过与标签通信并从标签中收集某些特定信息，读写器能够获知被标记物体的属性，例如保质期，也能够掌握周围环境状态，例如温度、湿度。

范围检测问题指的是：判断是否存在某些标签，其持有的数据值位于某个上界和下界之间。这些标签被称为目标标签。

更具体地，设标签集合为：

```text
T = {t1, t2, ..., tn}
```

其中每个标签 `ti`（`1 ≤ i ≤ n`）持有一个感兴趣的数据 `di`。本文假设 `di` 的值是一个正整数，每个整数代表某种属性值或某个特定的传感值。比如，如果用整数来编码 2020 年的日期，就可以约定：

- `1` 表示 `2020-01-01`
- `2` 表示 `2020-01-02`
- ...
- `366` 表示 `2020-12-31`

具体映射规则和目标范围由应用决定，不属于本文讨论范围。对于一个大型 RFID 系统，判断目标标签是否存在是困难且耗时的。本文的目标是在只使用商品化 RFID 设备的前提下，使检测时间最小化；不允许对 MAC 层通信协议或硬件做任何修改。高时间效率之所以重要，是因为在大型 RFID 系统中，它可以给用户提供更早的预警，并帮助在风险萌芽阶段减少潜在损失。

## 3. Exclusive Collection

Class1 Gen2（C1G2）协议是全球范围内的 UHF RFID 标准，它定义了商品化读写器和标签之间的物理交互和逻辑操作。

在 C1G2 中，读写器通过 inventory frame 来收集标签信息。一个 inventory frame 由若干个时隙组成。时隙是一个短时间窗口，标签可以在该窗口内与读写器通信。每个标签随机选择一个时隙，并在该时隙中向读写器发送自己的标签 ID，以及在需要时发送用户数据。

根据标签的选择，时隙可分为三类：

- singleton slot：恰有一个标签选择，可成功收集信息
- collision slot：有多个标签选择，会发生标签碰撞
- empty slot：没有任何标签选择

由于空槽没有标签、碰撞槽无法区分多个标签，因此只有单标签槽可用于收集标签信息。

如果要在 C1G2 范围内完成范围检测，最直接的方法就是对集合 `T` 中每个标签都执行 inventory。每读到标签 `ti` 的数据 `di`，就检查它是否落在给定范围内。如果是，就报告检测到目标事件；否则就继续读取另一个标签。这个方法直观且可靠，但问题在于：C1G2 一次只允许有效地读取一个标签，因此在范围检测中通常需要接近 `n` 次读取。

当目标标签远少于全部标签时，这种做法会极为低效。例如，有 1000 个标签而只有 1 个目标标签时，对整个集合逐个盘点显然很浪费。于是，减少 inventory 次数并尽快在大量标签中锁定目标标签，就成了提升性能的关键。

## 4. Selective Query

### 4.1 基本思想

Exclusive Collection 时间效率低，是因为读写器必须和每个标签通信，即便这个标签根本不是目标。如果能让大多数无关标签静默，就可以避免大量不必要的通信开销。

作者进一步观察实践中的被标记对象，得到两个事实。

第一，在一些场景中，待收集数据 `{di}` 的取值域大小远小于标签总数。例如，如果数据表示 2020 年的过期日期，那么可能的不同值只有 366 个，而大型超市中的商品种类和数量远超这个规模。根据抽屉原理，必然会有很多标签持有相同数据。

第二，不同标签可能具有相同属性或处于相似环境中。例如，同一批牛奶的保质期通常完全相同；邻近的带传感 RFID 标签往往会报告相同温度。

这两个观察启发出协议的核心思想：某个数据值只读取一次，而其余持有相同数据的标签都在之后保持静默。这样，读的次数就等于标签中不同数据值的个数，而在很多情况下它甚至比整个数据域规模还小得多。大量持有相同数据的标签将不会参加后续读取过程，从而显著节省通信时间。

接下来，作者讨论如何在 C1G2 范围内让标签静默并快速定位目标标签，这一方案称为 Selective Query（SQ）。

### 4.2 Selective Query（SQ）

#### 4.2.1 Select 功能

C1G2 在 inventory 之前定义了一个 `Select` 操作，它允许读写器选择一个特定标签子集，使这部分标签参与接下来的查询。换句话说，它可以让集合 `T - T'` 中的标签静默，只让 `T'` 参加后续过程。

`Select` 命令由 6 个字段组成。

第一组字段是：

- `MemBank`
- `Pointer`
- `Length`
- `Mask`

它们共同决定哪些标签算“匹配”。`MemBank` 指定要比较的存储区，可选四个银行之一：`MemBank-0`、`MemBank-1`、`MemBank-2`、`MemBank-3`。本文中的自定义数据存放在 `MemBank-3`，因此这里固定用 3。`Pointer` 指定比较的起始位置，`Length` 是掩码长度，`Mask` 是由应用决定的具体比特串。如果标签在指定 `MemBank` 中从 `Pointer` 开始、长度为 `Length` 的那段比特串与 `Mask` 相同，则该标签为匹配；否则为不匹配。

第二组字段是：

- `Target`
- `Action`

`Target` 指明 `Select` 要操作的是标签的哪个标志位，可以是 `SL` 标志，也可以是 inventoried flag。inventoried flag 是一个 1 比特状态，用于访问控制。换句话说，`Select` 的选择功能，本质上是：

1. 用掩码把感兴趣标签筛出来
2. 把匹配标签的标志设为某个值
3. 把不匹配标签的标志设为相反值或保持不变
4. 之后再对某种标志值的标签进行操作

`Action` 决定匹配和不匹配标签的标志如何变化。文中列出了 8 种动作，例如：

- `AB`：匹配标签设为 A，不匹配标签设为 B
- `A-`：匹配标签设为 A，不匹配标签不变
- `B-`：匹配标签设为 B，不匹配标签不变
- `BA`：匹配标签设为 B，不匹配标签设为 A

通过 `Target` 与 `Action` 的组合，读写器可以修改某一批标签的特定标志位。需要注意，C1G2 规定一个标签应支持 4 组 inventoried flag，对应 4 个 session，用于适应多读写器的独占读取场景。本文使用 session 2 中的 inventoried flag 来说明协议，其他标志可以类似处理。

#### 4.2.2 Query 功能

除了 `Select` 之外，C1G2 还定义了另一个常用命令 `Query`，它用于在一组处于特定标志状态的活跃标签上发起 inventory frame。

本文关注 `Query` 的三个字段：

- `Sel`
- `Session`
- `Target`

`Sel` 决定哪些标签会响应：

- `00` 和 `01` 表示响应前一条 `Select` 中被选中的标签
- `10` 表示响应 `~SL`
- `11` 表示响应 `SL`

由于本文不使用 `SL`，因此 `Sel` 始终设为 0。`Session` 选择使用哪个 session。本文使用 session 2，因此该字段固定为 2。`Target` 决定此次 inventory 面向标志为 `A` 的标签还是 `B` 的标签：

- `0` 表示标签的 inventoried flag 为 `A`
- `1` 表示标签的 inventoried flag 为 `B`

标签在被成功查询后，会把自己的 inventoried flag 从 `A` 翻转到 `B`，或从 `B` 翻转到 `A`。

#### 4.2.3 SQ 的设计

给定标签集合 `T` 及其数据 `{di}`，作者按数据值把标签划分为若干组：若且唯若两个标签 `ti` 和 `tj` 持有相同数据值，则它们属于同一组。设共有 `k` 个组，记为：

```text
G = {g1, g2, ..., gk}
```

SQ 的目标是：每个组只查询一次。因为如果来自同一组的多个标签都回复了相同数据，那只是通信开销的浪费。

要注意的是，这里的“组”只是为了分析方便而定义的；系统事先并不知道这些组的信息，也不知道标签 ID。协议的任务是在没有这些先验信息的情况下，依次把每个组的数据“采样”出来。

一般来说，SQ 最多执行 `k` 轮。

- 在第 1 轮，读写器先从当前活跃标签中读出一个标签 `ti` 及其数据 `di`
- 如果 `di` 落在给定目标范围内，就检测到了目标标签
- 否则，读写器利用 `di` 作为掩码，把和 `ti` 拥有相同数据的整组标签 `g'1` 静默

通过“1 次 query + 1 次 select”，就完成了对组 `g'1` 的采样。后续轮次重复同样操作。与第 1 轮不同的是，在第 `i` 轮静默组 `g'i` 时，必须保证之前已经静默的标签不会被重新激活。

下面作者把这一过程具体写成 Gen2 兼容命令。

一个 `Select` 命令记为：

```text
S(Target; Action; MemBank; Pointer; Length; Mask)
```

设活跃标签的 inventoried flag 为 `A`，静默标签的 flag 为 `B`。开始时，读写器先广播一条 `Select`，把所有标签都设成活跃：

```text
Flag <- AB : S(2, 0, 3, 0, 0, 0)
```

其中：

- `t = 2` 表示操作对象是 session 2 中的 inventoried flag
- `a = 0` 表示动作为 `AB`
- `(p, l, m) = (0, 0, 0)` 表示覆盖范围内全部标签都匹配

于是所有标签的 flag 都被置为 `A`。

之后，每一轮包含两步：一次 `Query` 和一次 `Select`。

用于对活跃标签发起 inventory 的命令为：

```text
Q(Sel, Session, Target)
```

如果要查询 session 2 中 flag 为 `A` 的活跃标签，则命令为：

```text
Q(0, 2, 0)
```

命令发出后，就执行一个 inventory frame。每个活跃标签随机选择一个槽并回复。读写器在单标签槽中收集一个标签的数据。设读到的标签为 `t'`，其数据为 `d'`。

- 若 `d'` 落入目标范围，则检测成功
- 否则，读写器通过如下 `Select` 将 `t'` 所在组静默：

```text
Flag <- B- : S(2, 5, 3, p', l', d')
```

其中 `p'` 和 `l'` 表示数据在存储区中的起始位置与长度；`a = 5` 表示动作为 `B-`，即匹配标签被置为 `B`，不匹配标签保持不变。换言之，所有数据值等于 `d'` 的标签都会被置为静默状态。

把上面各部分拼起来，SQ 的算法如下：

1. 先发 `Select`，把所有标签置为 `A`
2. 当仍存在 flag 为 `A` 的活跃标签时：
3. 发 `Q(0, 2, 0)`
4. 读取某个标签的数据 `d'`
5. 若 `d'` 在目标范围内，则报告检测到目标标签并结束
6. 否则发 `Select`，把所有持有数据 `d'` 的标签置为 `B`
7. 继续下一轮

当所有活跃标签都被静默，或者在某一轮检测到目标标签时，协议结束。

### 4.3 性能分析

接下来作者分析 SQ 的执行时间。设标签集合 `T` 根据数据值 `{di}` 被划分为 `k` 个组。

按算法过程，读写器首先发出一次 `Select`，激活视野中的所有标签。之后，系统对每个组重复执行：

- 隔离出该组的一个代表标签
- 收集其数据
- 用一次 `Select` 让拥有相同数据的整个组静默

如果系统中没有任何目标标签，那么读写器必须执行完整的 `k` 轮。此时 SQ 的执行时间 `T*` 为：

```text
T* = ts + k(tq + ts)
```

其中：

- `k` 是轮数
- `ts` 是发出一条 `Select` 的时间
- `tq` 是由一条 `Query` 触发的一次 inventory frame 的时间

在更一般的情况下，如果存在目标标签，则读写器不必真的执行 `k` 轮。只要某一轮发现目标标签，就可以立刻终止协议。假设协议在第 `i` 轮结束，这意味着：

- 前 `i-1` 轮都没有检测到目标标签
- 第 `i` 轮检测到了目标标签

设 `ρ` 是目标标签数量与总标签数 `n` 的比值。那么，在第 `i` 轮检测到目标标签、或协议在该轮结束的概率 `fi` 为：

```text
fi = (1 - μi)^(i-1) μi,    i < k
fi = (1 - μk-1)^(k-1),     i = k
```

其中 `μi` 表示第 `i` 轮检测到目标标签的概率：

```text
μi = ρ × n / (n - Σ|g'j|)
```

式中 `g'j` 表示前面已经采样过的组。也就是说，在剩余活跃标签集合中，目标标签比例会随着前面非目标组的移除而变化。

因此，SQ 的期望执行时间为：

```text
T = ts + (tq + ts) × Σ(i × fi)
```

显然，当目标标签数为 0 时，这一公式就退化为前面的 `T*`。由该表达式可以看到：SQ 的执行时间与组数 `k` 成正比，而不是与标签总数 `n` 成正比。在实际系统中，大量标签可能拥有相同属性或相似环境。例如，同一批商品的生产日期通常一致，邻近传感标签报告的湿度也可能相同。假设平均每组 50 个标签，也就是 `k = 0.02n`，那么 SQ 相比逐标签读取，理论上有机会带来一个数量级的性能提升。

## 5. Range Query

SQ 之所以能比 EC 快，是因为它有效地结合使用了 `Select` 与 `Query`：通过让许多持有相同数据的标签静默，大部分数据采集被避免了。然而，SQ 仍有两个问题。

第一，范围检测只需要一个二值答案：有还是没有。为了得到这个答案，却逐组收集每一组的数据，仍然是浪费。

第二，SQ 在“每组包含很多标签”的常见情况下表现很好，也就是 `k << n`。但在某些特殊情况下，如果 `k` 接近 `n`，那么轮数也会逼近 `n`，SQ 会退化回基本的逐标签读取。

于是作者提出新的问题：能否把 query 轮数进一步降到 1，而不管组数 `k` 和标签数 `n` 各是多少？答案是肯定的，这就是 Range Query（RQ）。

### 5.1 RQ 的设计

在数据库中，range query 是一种常见操作，用于检索那些值位于上下边界之间的记录。本文中的范围查询对应三类问题：

1. 是否存在数据值不大于某个下界的标签，记为 `LRQ`
2. 是否存在数据值大于某个上界的标签，记为 `URQ`
3. 是否存在数据值位于上下边界之间的标签，记为 `LURQ`

作者先详细介绍 `LRQ`，后两者可由它推广而来。

设下界为 `τ`。LRQ 的目标是检测是否存在某个标签，其数据 `di ≤ τ`。其基本思想是：先把所有目标标签（如果存在）从其他标签中分离出来，然后只对被选中的标签执行一次 inventory frame。只要有任何回复，就说明存在目标标签；如果没有回复，就说明没有。

关键挑战仍然是：在不知道标签 ID 的情况下，如何把所有 `di ≤ τ` 的标签从整个标签集合中分离出来。

由于 `Select` 命令可以选择一个子集参与后续查询，因此它非常适合用来做标签集合划分。为了完成 LRQ，需要让所有满足 `di ≤ τ` 的标签进入活跃状态，而让其他标签静默。

#### 5.1.1 一个直接但低效的 LRQ

直观方法是对区间 `[1, τ]` 中每个值逐个处理。

先看 `d' = 1`。读写器发一条 `Select`，掩码值是 1，动作是 `AB`。这样，数据值等于 1 的标签被视为匹配，它们的 inventoried flag 被置为 `A`；其余不匹配标签被置为 `B`。命令写作：

```text
Flag <- AB : S(2, a=0, 3, p', l', m=1)
```

其中：

- `a = 0` 表示动作 `AB`
- `(3, p', l')` 表示数据在用户存储区中的位置
- `m = 1` 表示只匹配数据值为 1 的标签

这样，值为 1 的标签被激活，其他标签静默。

对后续的每个数 `i ≤ τ`，读写器做类似的 `Select`，只是动作变成 `A-`，即匹配标签置为 `A`，其余标签保持不变：

```text
Flag <- A- : S(2, a=1, 3, p', l', i)
```

通过重复 `τ` 次，就能让所有数据属于 `[1, τ]` 的标签激活，而其他标签保持静默。最后只需再发一次：

```text
Query-A : Q(0, 2, 0)
```

对所有 flag 为 `A` 的标签执行 inventory。如果有任何回复，就说明存在目标标签。

这样，LRQ 把 inventory 次数从 `k` 降到了 1 次，相比 SQ 确实是巨大进步。但代价也很明显：需要 `τ` 次 `Select`。如果 `τ` 很大，就很不划算。例如，10000 个标签、`τ = 5000` 时，SQ 假设平均每组 100 个标签，则大概需要 `100` 次 select 和 `100` 次 inventory；而这种基本 LRQ 需要 `5000` 次 select 加 1 次 inventory，总体仍然很慢。根源在于它一次只处理一个数字。

#### 5.1.2 用子串掩码同时处理多个数字

为了解决这一问题，作者提出的核心思想是：不再用一个具体数值作为掩码，而是寻找一个公共二进制子串，使它能一次性选择多个数字，从而同时激活与这些数字对应的候选标签。

设阈值 `τ` 的二进制长度为 `l'`，则数据域是 `[1, 2^l']`。先看一个特殊情况：当

```text
τ = 2^x - 1
```

时，它的二进制形式为：

```text
00...0011...11
```

也就是左边 `l' - x` 位全为 0，右边 `x` 位全为 1。

这时，对于任何大于 `2^x - 1` 的数，右边 `x` 位在回卷后，左边的某位一定会变成非 0；而对于所有不超过 `2^x - 1` 的数，左边 `l' - x` 位必然都为 0。于是，仅用一个掩码“左边 `l' - x` 位全 0”，就可以把区间 `[1, τ]` 中所有数字对应的标签全部选出来。对应命令为：

```text
Flag <- AB : S(2, a=0, 3, p', l=l'-x, m=0)
```

这意味着：只要标签数据的左侧 `l' - x` 位全为 0，就会被激活。因此对于这类特殊阈值，一个 `Select` 就足够了。比如当 `x = 10` 时，基本 LRQ 原本需要 `1023` 次 select，而这里仅需 1 次。

#### 5.1.3 一般情形：用多个区间拼出 `[1, τ]`

真实情况下，`τ` 往往不是 `2^x - 1` 这种特殊形式。例如 `τ = 2^x` 时，不能只用一个掩码选出 `[1, 2^x]` 中所有数。为此，作者提出“递归地利用尽可能大的特殊子区间”。

比如先用前面的方式选出 `[1, 2^x - 1]`，然后再用一个额外 `Select` 单独激活数值 `2^x`：

```text
Flag <- A- : S(2, a=1, 3, p', l', m=2^x)
```

更一般地，对任意 `τ ∈ [1, 2^l']`，需要尽可能多地利用形如 `[1, 2^x - 1]` 这种“可由单个掩码整体表示”的最大区间。

若 `τ` 是奇数，其二进制可以写成：

```text
0...01 0...01 0...0 111...1
```

记 `d(τ)` 为最右端连续 1 的数量，记 `R(τ) = {ri}` 为二进制中所有 1 的位置集合，位置从右向左编号，最右边一位编号为 0。比如，`43 = 00101011₂`，于是：

```text
R(43) = {5, 3, 1, 0}
```

改进 LRQ 的基本做法是：先找出小于 `τ` 的最大 `2^x - 1`。因为这是 `[1, τ]` 中可以被单个 `Select` 一次性覆盖的最大子区间。

以 `τ = 43` 为例，小于它的最大 `2^x - 1` 是 `31`。于是先用 1 个 `Select` 覆盖 `[1, 31]`。然后继续递归处理剩余部分。第二次，用掩码选出 `[32, 39]`；第三次，再用一个掩码选出 `[40, 43]`。于是 `[1, 43]` 被拆成了三个连续区间：

1. `[1, 31]`
2. `[32, 39]`
3. `[40, 43]`

只需 3 条 `Select`：

```text
[1,31]   <- AB : S(2,0,3,p',3,m=000)
[32,39]  <- A- : S(2,1,3,p',5,m=00100)
[40,43]  <- A- : S(2,1,3,p',6,m=001010)
```

这样，区间 `[1,43]` 内所有数值对应的标签都被激活了。

如果 `τ` 是偶数，那么处理过程与奇数情况几乎完全一样，只是最后一轮不同：因为最右端是 0 而不是 1，所以最后剩下的是单个数字 `τ`，需要用一次单点 `Select` 处理。

#### 5.1.4 LRQ 算法

基于以上思路，文中给出 LRQ 算法：

1. 先计算 `R(τ)` 和 `d(τ)`
2. 如果 `τ = 2^(d(τ)) - 1`，说明它本身就是特殊形态，那么只用 1 个全 0 掩码就可选中 `[1, τ]`
3. 否则，先用最大 `2^x - 1` 对应的掩码选中 `[1, 2^x - 1]`
4. 对剩余部分，依次利用更短的前缀掩码继续选择后续子区间
5. 如果 `τ` 为奇数，则最后用一个掩码把最右侧连续 1 覆盖掉
6. 如果 `τ` 为偶数，则最后单独选择数字 `τ`
7. 最后发起一次 `Query-A : Q(0, 2, 0)`，只对被激活标签做 inventory
8. 若有任何回复，则返回 yes，否则返回 no

这样，所有数据属于 `[1, τ]` 的标签都会处于 `A`，其他标签为 `B`。最后一条 `Query` 只要收到一个回复，就说明目标存在。

### 5.2 URQ 与 LURQ

前面已经详细介绍了 `LRQ`。另外两种查询可以很容易地从它推广出来。

#### 5.2.1 URQ

URQ 的问题是：是否存在某些数据值大于某个上界。

做法很简单：把 LRQ 算法中 `Select` 的动作全部反过来即可：

- `AB` 改成 `BA`
- `A-` 改成 `B-`

这样，大于上界的标签会被设为 `A`，其余标签会被设为 `B`。最后再发：

```text
Query-A : Q(0, 2, 0)
```

如果收到回复，则说明存在目标标签。

#### 5.2.2 LURQ

LURQ 要回答的问题是：是否存在数据位于区间 `(τL, τU]` 内。

作者的方法是把 LRQ 跑两次，但做一个小修改。

第一次：执行 `LRQ(τU)` 的“选择阶段”而不发最后的 query。这样，所有数据属于 `[1, τU]` 的标签被设为 `A`，其余设为 `B`。

第二次：执行 `LRQ(τL)`，但把所有动作改为 `B-`。这样，`[1, τL]` 中的标签被进一步改成 `B`。

于是最终只有 `(τL, τU]` 内的标签保持 `A`。最后对这些标签发一次 query 即可。文中把这两步写为：

```text
[1, τU] <- A : 执行 LRQ(τU) 的第 1-14 行
[1, τL] <- B : 执行 LRQ(τL)，但全部动作改为 B-
```

### 5.3 Mask Combination

作者通过查阅商品化 RFID 读写器的数据手册并结合实际实验发现，这些读写器允许一条 `Select` 命令中携带多个掩码。例如：

- Impinj R420 最多支持 2 个 mask
- ALR 9900+ 与 ALR F800 最多支持 4 个 mask

利用这一功能，可以把多个原本需要分别发出的基础 `Select` 填进同一条命令中，从而把多次选择压缩成一次，大幅节省通信开销。这就是文中的 mask combination 技术。

### 5.4 性能分析

下面分析 LRQ 的执行时间，URQ 和 LURQ 可以在此基础上轻松得到。

给定下界 `τ`，LRQ 的通信开销由两部分组成：

1. 读写器为覆盖区间 `[1, τ]` 发出的若干条 `Select`
2. 一条 `Query` 命令以及一次 inventory frame，用于检查这些候选标签中是否有回复

根据算法，`Select` 数量由 `|R(τ)|` 和 `d(τ)` 决定，其中：

- `|R(τ)|` 是 `τ` 的二进制中 1 的个数
- `d(τ)` 是最右端连续 1 的个数

因此，所需 `Select` 总数为：

```text
f(τ) = |R(τ)| - d(τ) + 1
```

这是一个非常显著的下降，因为 `|R(τ)|` 小于 `ceil(log2 τ)`，而后者又远小于 `τ` 本身。举例来说，当 `τ = 5000` 时，LRQ 只需 `6` 次 `Select`，远小于基本 Range Query 中需要的 `5000` 次。再考虑 mask combination 后，所需 `Select` 数还会继续下降。

于是 LRQ 的执行时间可写为：

```text
T(τ) = ceil(f(τ) / ω) × ts + tq
```

其中：

- `ω` 是一条 `Select` 可容纳的掩码数
- `ts` 是发出一条 `Select` 的时间
- `tq` 是一条 `Query` 加一次 inventory frame 的时间

由此可见，LRQ 的执行时间随着 `Select` 数量骤减而大幅下降。例如，10000 个标签、`τ = 5000` 时，如果采用 Alien 读写器且 `ω = 4`，那么 LRQ 只需要 `2` 条 `Select` 再加一次 inventory，就能完成检测；相比 SQ 假设平均每组 100 标签时需要 `100` 次 select 和 `100` 次 inventory，这显然快得多。

对 URQ 而言，执行时间与 LRQ 完全相同。对 LURQ，则因为需要两轮选择，所以执行时间为：

```text
T(τL, τU) = ceil((f(τL) + f(τU)) / ω) × ts + tq
```

## 6. 实验评估

### 6.1 实验设置

作者在商品化 RFID 读写器和标签上实现并评估了上述协议。实验中使用了来自两家主流 RFID 厂商 Impinj 与 Alien 的 4 种 UHF RFID 读写器。每个读写器连接一个增益为 `8.5 dBic`、工作频率约 `920 MHz` 的定向天线。

为了更贴近真实系统并观察协议的实际效果，实验环境搭建在图书馆中，最多使用 `1000` 个商品化标签。

实验首先考察文中使用到的读写器是否支持：

- `Select`
- inventory

结果显示 4 款读写器都支持这两类操作。但是：

- Impinj 读写器只向用户提供字符级掩码能力，也就是说，mask 的起始位置必须是某个字符的第一个 bit，即第 `4i` 位
- Alien 读写器，例如 `ALN-F800`、`ALN-9900+`，能够提供按位级别的 mask

而本文协议正需要位级掩码，因此最终选用 `ALN-F800` 完成实现与评估。作者还指出：位级掩码本身是 C1G2 规范里定义的，Impinj 的限制更可能是没有把底层能力完全开放给用户。

### 6.2 Select 数量

第 5 节中，作者已经从理论上分析了 RQ 需要的 `Select` 数量。实验先对这部分进行验证。

#### 6.2.1 LRQ / URQ 的 Select 数

图 2(a) 中，作者让阈值 `τ` 的二进制长度从 1 变化到 16，并分别考察两种情况：

- 不使用掩码组合（RQ）
- 使用掩码组合（RQ-W）

由于 Alien 读写器允许一条 `Select` 命令携带 4 个 mask，因此式中的 `ω` 设为 4。

实验结果表明，随着 `τ` 的比特长度增大，最大和平均 `Select` 数都近似线性上升。这是符合直觉的，因为阈值越长，二进制中出现更多 1 的概率越高。但这并不意味着“阈值越大，Select 数就一定更多”，因为真正决定它的是：

- 二进制中 1 的个数
- 右端连续 1 的个数

例如：

- 当 `τ = 20000` 时，LRQ 只需要 `6` 次 `Select`
- 当 `τ = 2000` 时，反而需要 `7` 次

说明 Select 数取决于二进制结构，而不是十进制大小本身。加入 mask combination 之后，这一数量会进一步下降。实验结果验证了 LRQ 与 URQ 确实能通过子串掩码和掩码组合，把 `Select` 数压到很小。

#### 6.2.2 LURQ 的 Select 数

图 2(b) 研究了 LURQ 在不同 `τL` 和 `τU` 下的 `Select` 数，二者长度都在 1 到 16 位之间，且 `τU > τL`。

对每一组具体长度组合，作者绘制了 LURQ 覆盖区间 `(τL, τU]` 所需平均 `Select` 数。结果显示：

- 最大 `Select` 数小于 17
- 若考虑 mask combination，则可进一步下降到约 5

这说明，无论查询区间如何变化，LURQ 都具有很好的时间效率。

### 6.3 时间效率

高时间效率对于范围检测至关重要，因为它直接关系到能否及时发出预警并帮助用户降低潜在风险。由于此前没有工作研究“C1G2 范围内的范围检测”，因此作者把 Exclusive Collection（EC）作为基线。

#### 6.3.1 三种协议整体对比

图 3 比较了 SQ、RQ（实验中使用 LRQ）以及基线 EC 在 3 个场景下的执行时间。RQ 中的阈值或区间在数据域中随机选择。

三个场景为：

- 场景 1：`n = 500, l' = 16, ρ = 0.1%`
- 场景 2：`n = 1000, l' = 16, ρ = 0.1%`
- 场景 3：`n = 1000, l' = 16, ρ = 0`

其中：

- `n` 是标签数
- `l'` 是数据二进制长度
- `ρ` 是目标标签数与 `n` 的比值

在 SQ 中，作者假设平均每组 20 个标签。对于场景 2，由于一个天线无法覆盖 1000 个标签，因此使用了两个天线。场景 3 中 `ρ = 0`，表示系统中不存在任何目标标签。

以场景 2 为例：

- EC：`9.6s`
- SQ：`3.2s`
- LRQ：`0.33s`

因此，LRQ 相比 EC 提升了近 `30×`。另外两个场景也得到相同结论：RQ 最好，SQ 次之，EC 最差。

#### 6.3.2 标签数 `n` 的影响

图 4(a) 研究标签数 `n` 对 EC、SQ、RQ 执行时间的影响。实验中固定：

- `l' = 16`
- `ρ = 0.1%`

让 `n` 从 100 增加到 1000。当 `n > 500` 时，实验使用两个天线，因为每个天线最多覆盖约 500 个标签。SQ 假设平均每组 20 个标签。RQ 采用最坏情况，即需要 4 次 `Select`。

结果显示：

- EC 的执行时间随 `n` 增长而线性上升，因为它必须和每个标签单独通信
- SQ 也随着 `n` 上升，但绝对值更小，因为它能静默大量同值标签
- RQ 基本保持稳定，因为其代价主要取决于阈值结构，而不是标签总数

例如，当 `n = 500` 时：

- EC：`4.8s`
- SQ：`1.6s`
- RQ：`0.17s`

RQ 相比 EC 带来约 `28×` 的提升。

#### 6.3.3 目标标签比例 `ρ` 的影响

图 4(b) 研究 `ρ` 对执行时间的影响。固定：

- `n = 1000`
- `l' = 16`

让 `ρ` 从 0 增加到 1%，步长为 0.1%。

结果表明：

- EC 和 SQ 的执行时间随着 `ρ` 增加而下降，因为目标越多，协议越可能在早期就检测到目标并提前结束
- RQ 的执行时间基本保持稳定，因为它无论目标比例多少，都要先选择整个目标区间再统一发起一次查询

#### 6.3.4 数据长度 `l'` 的影响

图 4(c) 研究数据二进制长度 `l'` 的影响。固定：

- `n = 1000`
- `ρ = 0.1%`

让 `l'` 取 `16、32、48`。由于读写器按字进行用户数据读取，因此以 16 位为步长。

结果显示：

- EC 和 SQ 都随着 `l'` 增加而略微变慢，因为读写器需要读取更长的数据
- RQ 也会因为边界变长而增加一些 `Select` 数量

即便如此，RQ 相比基线依然优势明显。

## 7. 相关工作

范围检测本质上是在判断是否存在标签，其数据值位于某个上下边界之间，因此可以视为信息收集问题的一个特例。信息收集一直是 RFID 研究的重要方向。这里的“信息”既可以是标签 ID，也可以是商品属性，或者在带传感能力的 RFID 系统中采集到的传感数据，例如冷链中的温度。

为了实现高效率，早期工作使用：

- Aloha 类协议
- 树型协议

来高效识别标签。

对 Aloha 类协议而言，基本思想是让标签在不同时间槽发送数据。每个标签随机选择一个槽，只有恰好被一个标签选中的槽才可用于收集 ID，从而有效避免标签碰撞。

树型协议则利用标签 ID 的动态前缀，把标签集合不断拆分为更小子集，直到每个子集中只剩一个标签。该过程不断迭代，最终完成所有标签识别。

对于其他信息收集问题，也有不少改进工作。例如：

- Chen 等提出多哈希信息收集协议，用多个哈希函数避免槽分配中的碰撞
- Yue 等研究多读写器 RFID 系统，并用 Bloom Filter 判断哪些标签被每个读写器覆盖
- Qiao 等提出轮询辅助协议，以节能方式从标签子集中收集信息
- Liu 等提出增量轮询协议，把轮询向量从 96 位压缩到不足 2 位，大幅节省轮询开销
- Liu 等研究多类别 RFID 系统中的类别信息收集，不再逐个标签重复询问，而是通过快速缩放与隔离标签，以较小开销从每个类别中采样
- Liu 等还研究了“range query”问题，但那一工作是要判断每个标签数据落在哪个区间，本质上是在粗粒度收集每个标签数据，目标与本文不同
- Chen 等设计 differential Bloom filter（DBF）用于自动检测 RFID 异常事件，例如缺失标签、未知标签、克隆标签，而不是针对标签存储器中的具体数据值

尽管这些工作各有推进，但它们无法直接在商品化 RFID 系统中工作，因为它们与 C1G2 不兼容。原因在于，这些协议往往依赖一些理想化假设，例如标签端哈希或构建专用过滤器，而这些都不是 C1G2 所支持的能力。

## 8. 结论

本文研究了商品化 RFID 系统中的范围检测问题。通过深入利用 C1G2 兼容命令，作者提出了两种定制方案：

- Selective Query（SQ）
- Range Query（RQ）

SQ 根据数据值把标签划分为若干组，并且每组只查询一次，从而把 inventory 次数降低到组数级别。RQ 进一步把 inventory 次数降到恰好 1 次：它通过精心设计的一组 `Select` 先把目标标签（如果存在）从整个标签集合中快速分离出来，然后只发起一次查询。

作者在商品化 RFID 系统中实现了这两种协议，不需要任何硬件修改。大量实验表明，相比基线方案，RQ 能将时间效率提升一个数量级以上。
