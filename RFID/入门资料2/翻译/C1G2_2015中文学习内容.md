# C1G2_2015 中文学习内容

本文档用于配合文件夹中的两篇 RFID 论文学习：

- `Fast RFID Grouping Protocols`：代表较早期的 RFID 协议设计思路，论文中的部分能力不完全受商品化标准约束。
- `Time-efficient Range Detection in Commodity RFID Systems`：课题组在 C1G2 标准能力内完成范围检测，重点体现“标准已有功能如何被组合成新协议”。

学习 C1G2_2015 不需要逐页精读标准。更重要的是理解它规定了读写器和标签之间“能做什么、怎么做、受到什么限制”。下文重点翻译并整理与协议设计最相关的内容：标签存储区、flag/session、Select、Query、inventory frame、slot、RN16/ACK、Read/Write，以及这些机制如何支撑两篇论文。

## 1. 先把 C1G2 放在系统图景里

C1G2 即 EPCglobal Class-1 Generation-2 UHF RFID 空中接口协议，工作在 UHF RFID 的 860 MHz 到 960 MHz 频段。它规定的是读写器与无源或半无源标签之间的底层交互方式，包括：

- 读写器如何给标签供能并发送命令；
- 标签如何反向散射回复；
- 多个标签同时存在时如何避免或处理碰撞；
- 标签如何被筛选、盘点、读取、写入、锁定或杀死；
- 标签有哪些逻辑状态、存储区和标志位。

从论文学习角度，可以先把 C1G2 理解为一个“受限但可编程”的标签集合操作接口：

1. `Select`：按内存比特掩码筛选标签，并修改标签的状态标志。
2. `Query / QueryRep / QueryAdjust`：让满足状态条件的标签进入一轮盘点。
3. `ACK`：确认某个单标签回复，获得其 EPC/PC/CRC 等信息。
4. `Req_RN / Read / Write`：对已经识别出的单个标签执行访问操作。

第二篇论文 `Range Detection` 的核心价值，正是把这些标准命令组合起来，避免逐个读取所有标签。

## 2. C1G2 的三类核心操作

标准把读写器管理标签群体的操作分成三类。

### 2.1 Select：选择标签子集

`Select` 用于在盘点前选择某一部分标签。它不会让标签回复，也不会读取数据；它只让标签根据某段内存是否匹配指定掩码，去修改自己的 `SL` 或某个 session 的 inventoried flag。

可以理解为：

```text
读写器：满足这个内存条件的标签，把某个 flag 改成 A/SL；不满足的改成 B/~SL 或保持不变。
标签：自己比较内存并改状态，不回传任何数据。
```

这就是 C1G2 中最像“批量条件筛选”的能力。

### 2.2 Inventory：盘点标签

Inventory 是识别标签的过程。读写器用 `Query` 开始一个 inventory round，让符合 `SL` 与 inventoried flag 条件的标签参与。参与标签随机选择时隙回复，因此一轮里会出现三类时隙：

- `singleton slot`：只有一个标签回复，读写器可以识别它。
- `collision slot`：多个标签同时回复，普通情况下读写器无法区分。
- `empty slot`：没有标签回复。

标签首先回复的是 `RN16`，读写器如果成功收到一个 RN16，就用 `ACK` 回显该 RN16，标签再回传 PC、EPC 和 CRC。这个两步过程避免了多个标签同时直接发送长 EPC。

### 2.3 Access：访问单个标签

Access 是在标签被识别之后，对单个标签执行读取、写入、锁定、认证等操作。典型流程是：

1. 先通过 inventory 让某个标签进入 `acknowledged` 状态。
2. 读写器发送 `Req_RN`。
3. 标签返回新的 `RN16`，该值成为后续访问命令使用的 `handle`。
4. 读写器用带 handle 的 `Read`、`Write` 等命令访问标签。

因此，`Read/Write` 通常不是群体操作，而是针对已经 singulate 出来的单个标签。

## 3. 标签内存：Select 和 Read/Write 的对象

C1G2 标签逻辑上分为四个 memory bank。不同命令通过 `MemBank` 字段指定要操作哪个存储区。

| MemBank 二进制值 | 中文名称 | 主要内容 | 学习重点 |
|---|---|---|---|
| `00` | Reserved memory | Kill password、Access password | 安全、访问权限、Kill/Access 命令相关 |
| `01` | EPC memory | StoredCRC、StoredPC、EPC、可选 XPC | 盘点后 ACK 返回的 EPC 主要来自这里 |
| `10` | TID memory | 标签类型、厂商、型号、可选特性信息 | 用于识别标签能力，不是应用数据的主要位置 |
| `11` | User memory / File_0 | 用户自定义数据，可选 | 第二篇范围检测通常把待检测数据放在这里 |

需要特别注意两点。

第一，`Select` 的 `Pointer` 是按 bit 寻址的；而 `Read/Write` 的 `WordPtr` 是按 16-bit word 寻址的。论文里用掩码比较某段数据时，实际对应的是 bit 级偏移。

第二，User memory 是可选的。论文中的范围检测把保质期、温度等应用数据放入 User memory，这在实验或特定标签上可行，但标准本身并不保证所有标签都有 User memory。

## 4. Session、inventoried flag 与 SL flag

理解 C1G2 的 Select/Query，关键是理解 flag。

### 4.1 四个 session 与 inventoried flag

C1G2 定义 4 个 session：`S0`、`S1`、`S2`、`S3`。每个标签在每个 session 中都有一个独立的 inventoried flag，取值为 `A` 或 `B`。

可以把 inventoried flag 理解为每个 session 里的二值状态：

- `A`：可被某次 Query 选作参与对象；
- `B`：也可被 Query 选作参与对象，但通常作为“已处理/静默”的另一侧状态使用。

读写器发 `Query` 时会指定：

- 使用哪个 session；
- 让 flag 为 `A` 的标签参与，还是让 flag 为 `B` 的标签参与。

当一个标签在某个 session 中被成功 singulate 后，后续 `Query`、`QueryRep` 或 `QueryAdjust` 可能使它的 inventoried flag 在 `A/B` 之间翻转。这个机制用于避免同一标签在同一轮中反复被读取。

### 4.2 为什么论文常用 S2

第二篇论文示例中常用 session 2，即 `S2`。原因不是 S2 有特殊计算能力，而是它的 flag 持久性较适合控制标签状态：标签有电时 S2/S3 flag 可保持，短时间断电后也有一定持久性。相比之下，S0 更像即时盘点，断电后不保持；S1 的保持时间较短。

对论文学习来说，可以把 `S2` 当成一个较稳定的“临时标记空间”。

### 4.3 SL flag

`SL` 是另一个 selected flag，不属于任何 session，所有 session 共用。`Select` 可以 assert 或 deassert SL；`Query` 可以选择盘点 `SL` 标签、`~SL` 标签，或忽略 SL。

论文中不一定必须使用 SL。第二篇范围检测主要使用 session 2 的 inventoried flag 来控制活跃/静默状态。

## 5. 标签状态机：理解命令为什么有先后顺序

C1G2 标签有若干逻辑状态。学习时不需要背状态转移表，但要理解每个状态的作用。

| 状态 | 中文理解 | 典型触发/作用 |
|---|---|---|
| `ready` | 已上电但未参与当前盘点 | 标签进入读写器场后通常从这里开始 |
| `arbitrate` | 正在参与盘点但当前槽计数器不为 0 | 等待后续 `QueryRep` 递减槽计数器 |
| `reply` | 轮到自己回复 RN16 | 槽计数器变为 0 后进入 |
| `acknowledged` | RN16 被 ACK 成功确认 | 标签回传 EPC，可进入访问流程 |
| `open` | 已进入访问但权限较低 | Access password 非零时常先进入 |
| `secured` | 访问权限更高 | 可执行更多访问命令 |
| `killed` | 永久失效 | Kill 成功后进入，不再响应 |

这解释了为什么 `Read/Write` 不能像 `Select` 那样直接广播给一群标签：访问类命令需要标签已经被识别并持有正确 handle。

## 6. Select 命令：C1G2 中最重要的集合筛选工具

### 6.1 Select 做什么

`Select` 根据标签某个 memory bank 中的一段比特是否等于 `Mask`，把匹配和不匹配标签的 flag 改成不同状态。它的核心字段如下。

| 字段 | 含义 | 学习重点 |
|---|---|---|
| `Target` | 指定要修改 SL，还是某个 session 的 inventoried flag | `010` 表示 session 2 的 inventoried flag |
| `Action` | 指定匹配/不匹配标签如何改 flag | 决定置 A、置 B、保持、翻转 |
| `MemBank` | 指定掩码作用的存储区 | `11` 常用于 User memory |
| `Pointer` | 掩码比较起始 bit 地址 | bit 寻址，不是 word 寻址 |
| `Length` | 掩码长度，最多 255 bit | 决定比较多少位 |
| `Mask` | 标签本地内存要匹配的比特串 | 用于精确值、前缀、子串等匹配 |
| `Truncate` | 是否让 ACK 回复截断 EPC | 学论文时通常不是重点 |

### 6.2 Target 的常用取值

| Target | 作用对象 |
|---|---|
| `000` | S0 inventoried flag |
| `001` | S1 inventoried flag |
| `010` | S2 inventoried flag |
| `011` | S3 inventoried flag |
| `100` | SL flag |

第二篇论文里的 `S(2, action, 3, pointer, length, mask)` 可理解为：操作 session 2 的 inventoried flag，在 User memory 中按掩码筛选。

### 6.3 Action 的学习版翻译

Action 一共有 8 种。为了方便记忆，可以用 `A` 表示置为活跃侧，`B` 表示置为静默侧，`-` 表示不变。

| Action 二进制 | 常用记法 | 匹配标签 | 不匹配标签 | 用法理解 |
|---|---|---|---|---|
| `000` | `AB` | 置为 A | 置为 B | 用一个条件把集合切成“匹配/不匹配”两边 |
| `001` | `A-` | 置为 A | 不变 | 把匹配标签加入活跃集合 |
| `010` | `-B` | 不变 | 置为 B | 保留匹配侧，压制不匹配侧 |
| `011` | `invert-` | 翻转 | 不变 | 只翻转匹配标签 |
| `100` | `BA` | 置为 B | 置为 A | 与 `AB` 方向相反 |
| `101` | `B-` | 置为 B | 不变 | 把匹配标签从活跃集合中移除 |
| `110` | `-A` | 不变 | 置为 A | 把不匹配标签加入活跃集合 |
| `111` | `-invert` | 不变 | 翻转 | 只翻转不匹配标签 |

这张表是理解 `Range Detection` 的关键。例如：

- 先用 `AB` 把某个条件匹配的标签置为 A，其他置为 B。
- 后续再用 `A-` 把其他条件匹配的标签也加入 A，从而实现多个条件的并集。
- 用 `B-` 可以把某一类标签从 A 中移除，用于逐组静默。

### 6.4 连续 Select 可以实现并集、交集、取反

单条 `Select` 只能比较一个掩码条件，但连续多条 `Select` 可以组合出更复杂的集合操作。

- 并集：先把所有标签置为 B，再对每个目标条件执行 `A-`，把匹配标签逐步加入 A。
- 交集：先让候选集合为 A，再对不满足某条件的标签置 B，逐步缩小集合。
- 取反：用 `AB` 或 `BA` 直接把匹配侧与不匹配侧分开。

第二篇论文的 `substring masking` 和 `mask combination` 本质上就是把范围条件转换成少量可由 `Select` 表达的掩码条件。

### 6.5 Select 的常见误区

`Select` 不会读出标签数据。它只让标签自己比较本地内存并改变状态。

`Select` 的匹配不是数值比较，而是比特串相等比较。所谓范围查询，需要把数值范围拆成若干比特模式组合，不能直接让标签执行“大于/小于”。

`Select` 不解决碰撞。它只是减少后续参与 `Query` 的标签数量；真正发生回复和碰撞管理的是 inventory 过程。

## 7. Query 与 inventory frame

### 7.1 Query 做什么

`Query` 开始一轮 inventory round，并规定哪些标签参与。核心字段如下。

| 字段 | 含义 | 学习重点 |
|---|---|---|
| `DR` | Tag-to-reader 反向链路频率参数 | 性能计算时影响时长，一般不是协议逻辑重点 |
| `M` | 标签回复调制/编码参数 | 影响速率 |
| `TRext` | 是否使用扩展前导 | 影响回复格式 |
| `Sel` | 是否基于 SL 选择标签 | `00/01` 忽略 SL，`10` 选 `~SL`，`11` 选 `SL` |
| `Session` | 选择 S0/S1/S2/S3 | 与 inventoried flag 搭配 |
| `Target` | 选择 flag 为 A 还是 B 的标签 | `0` 表示 A，`1` 表示 B |
| `Q` | 设定本轮槽数规模 | 标签随机从 `0` 到 `2^Q - 1` 中选槽 |

如果论文写 `Q(0, 2, 0)`，可以理解为：忽略 SL，使用 session 2，只让 S2 inventoried flag 为 A 的标签参与。

### 7.2 Q 参数与 slot

参与标签收到 `Query` 后，会随机选择一个 `Q` 位值作为 slot counter：

```text
slot counter ∈ [0, 2^Q - 1]
```

- 如果抽到 0，标签立即进入 `reply` 并发送 RN16。
- 如果不是 0，标签进入 `arbitrate`，等待后续 `QueryRep`。
- 每次收到匹配 session 的 `QueryRep`，slot counter 减 1。
- 当计数器减到 0，标签发送 RN16。

因此，一轮 inventory 可以看成有约 `2^Q` 个槽。`Q` 太小会导致碰撞多，`Q` 太大会导致空槽多。许多 RFID 协议优化，本质上都在减少无效槽、碰撞槽或不必要参与的标签数。

### 7.3 三类槽与两篇论文的关系

| 槽类型 | 标准含义 | 对论文理解的作用 |
|---|---|---|
| empty slot | 没有标签回复 | 时间浪费；协议希望减少 |
| singleton slot | 一个标签回复 | 可读出 RN16，进一步 ACK 得到 EPC/数据 |
| collision slot | 多个标签回复 | 常规盘点无法直接读出；第一篇论文尝试利用碰撞思想，第二篇尽量避免无关碰撞 |

`Fast RFID Grouping Protocols` 中的 CCG 强调“利用碰撞”来让多个标签同时获得信息。这个思路比较偏协议创新，不完全等同于商品化 C1G2 设备可直接支持的能力。

`Range Detection` 则相反：它尽量用 `Select` 先压缩候选集合，使最后的 `Query` 只对目标标签集合执行；只要存在任意回复，就说明范围内有目标标签。

### 7.4 QueryRep 与 QueryAdjust

`QueryRep` 用来推进当前 inventory round：它不改变 `Q`，只让参与标签递减 slot counter。

`QueryAdjust` 用来调整当前轮的 `Q`，可以增大、减小或保持 `Q`，然后让标签重新选择槽。

理解上可以简化为：

- `Query`：开始新一轮。
- `QueryRep`：进入下一槽。
- `QueryAdjust`：根据当前碰撞/空槽情况调整槽规模。

## 8. RN16、ACK 与 EPC 回复

C1G2 中标签不是一上来就直接发完整 EPC，而是先发一个 16 bit 随机数 `RN16`。

典型单标签识别过程：

1. 读写器发 `Query`。
2. 某个标签选中当前槽，发送 `RN16`。
3. 读写器发 `ACK(RN16)`。
4. 标签确认 ACK 中的 RN16 与自己刚发的一致，然后回传 PC、EPC 和 CRC。
5. 标签进入 `acknowledged` 状态。

这种设计的作用是：

- 短回复优先，降低碰撞代价；
- ACK 确认具体标签，避免多个标签同时发送长 EPC；
- 后续访问命令可以从 acknowledged 状态继续进行。

如果多个标签同时发 RN16，读写器看到的是碰撞。普通商品设备通常不能从碰撞中恢复多个标签的完整身份，除非底层接收能力或协议另有扩展。

## 9. Read、Write 与用户数据

### 9.1 Read

`Read` 用于读取标签某个 memory bank 的一段内容。它的核心字段是：

| 字段 | 含义 |
|---|---|
| `MemBank` | 选择 Reserved、EPC、TID 或 User |
| `WordPtr` | 起始 word 地址，按 16 bit word 寻址 |
| `WordCount` | 读取多少个 16 bit word |
| `RN` | 当前访问会话中的 handle |
| `CRC` | 命令校验 |

`Read` 的前提是标签已经被 singulate，并进入 open 或 secured 状态。也就是说，它适合读取某个已识别标签的数据，不适合直接对所有标签做批量范围判断。

### 9.2 Write

`Write` 每次写入一个 16 bit word。写入前读写器需要通过 `Req_RN` 获取新的 RN16，并用该 RN16 对待写数据做异或覆盖编码。这样可以减少明文直接暴露在空口上的风险。

对两篇论文来说，`Write` 不是核心优化对象，但它解释了应用数据如何预先写入标签。例如，保质期、温度编码、类别信息等可以被写入 User memory，之后再由 `Select` 或 `Read` 使用。

### 9.3 为什么范围检测不直接 Read 所有标签

逐个 `Read` 的成本很高，因为每个标签都要经历识别和访问流程。第二篇论文的核心动机就是避免这种方式：

- 不逐个读取所有 `di`；
- 先用 `Select` 的掩码匹配能力让非目标标签静默；
- 再用一次或少数几次 `Query` 判断目标集合是否为空。

## 10. 标准命令速查

| 命令 | 类型 | 是否必需 | 作用 |
|---|---|---|---|
| `Select` | Select | 必需 | 按内存掩码修改 SL 或 inventoried flag |
| `Query` | Inventory | 必需 | 开始一轮盘点，设定 session、target、Q |
| `QueryRep` | Inventory | 必需 | 推进当前轮，下一个槽 |
| `QueryAdjust` | Inventory | 必需 | 调整 Q 并继续当前轮 |
| `ACK` | Inventory | 必需 | 确认 RN16，让标签回传 EPC 等 |
| `NAK` | Inventory | 必需 | 让标签回到 arbitrate，不改变 inventoried flag |
| `Req_RN` | Access | 必需 | 获取新的 RN16/handle，进入访问流程 |
| `Read` | Access | 必需 | 读取指定 memory bank 的若干 word |
| `Write` | Access | 必需 | 写入一个 16 bit word |
| `Lock` | Access | 必需 | 设置内存或密码的锁定状态 |
| `Kill` | Access | 必需 | 永久禁用标签 |
| `BlockWrite` / `BlockErase` | Access | 可选 | 批量写入或擦除 |
| `Authenticate` / `Challenge` | Security | 可选 | 安全认证相关 |

学习两篇论文时，最重要的是前 8 个命令，尤其是 `Select`、`Query`、`QueryRep`、`ACK`、`Req_RN`、`Read`。

## 11. 如何用 C1G2 描述第二篇论文的 SQ

Selective Query 的思想是：同一个数据值只采样一次。假设活跃标签用 S2 inventoried flag 的 `A` 表示，静默标签用 `B` 表示。

### 11.1 初始化

先把所有标签置为 A：

```text
Select(Target=S2, Action=AB, MemBank=User, Pointer=0, Length=0, Mask=空)
```

`Length=0` 可理解为所有标签都匹配，于是所有标签进入 A。

### 11.2 每轮查询

读写器对 A 标签执行：

```text
Query(Sel=忽略SL, Session=S2, Target=A)
```

如果某个标签被读到，得到它的数据值 `d'`：

- 若 `d'` 在目标范围内，则检测成功。
- 若不在范围内，则用 `Select` 把所有数据等于 `d'` 的标签置为 B：

```text
Select(Target=S2, Action=B-, MemBank=User, Pointer=p, Length=l, Mask=d')
```

这样，所有拥有同一数据值的标签都不再参与后续 Query。SQ 的效率取决于不同数据值的数量，而不是标签总数。

## 12. 如何用 C1G2 描述第二篇论文的 RQ

Range Query 更进一步：范围检测只需要回答“是否存在目标标签”，不一定要读出目标标签是谁。因此可以把目标标签集合先筛出来，再只做一次 inventory。

### 12.1 关键限制

C1G2 的 `Select` 只能做“某段比特是否等于 Mask”，不能直接做数值大小比较。因此，范围条件必须被转换成若干个比特串匹配条件。

例如，判断 `di <= τ` 时，可以把满足条件的二进制数集合拆成若干前缀或子串模式。每个模式用一条 `Select` 表达，多个模式用 `A-` 累加到 A 集合中。

### 12.2 RQ 的抽象流程

```text
1. 先把所有标签置为 B。
2. 把范围条件转换成若干个掩码模式。
3. 对每个模式发 Select(Action=A-)，把匹配标签加入 A。
4. 发 Query(Session=S2, Target=A)。
5. 如果有任何 RN16/EPC 回复，则范围内存在目标标签；如果没有回复，则不存在。
```

RQ 的优势在于最后只需要一次 inventory。`Select` 的数量与掩码分解有关，而不直接随标签数增长。

### 12.3 substring masking 和 mask combination

`substring masking` 的思想是：不必总是匹配完整数据，可以只匹配能代表某个范围片段的关键子串或前缀。这样一条 `Select` 能覆盖一批目标值。

`mask combination` 的思想是：把多个可合并的掩码条件组合起来，减少 `Select` 条数。

它们利用的都是同一个标准能力：`Pointer + Length + Mask` 对标签内存的一段比特做相等匹配。

## 13. 第一篇论文与 C1G2 的关系

`Fast RFID Grouping Protocols` 关注的是：后端已经知道标签分组结果，如何尽快把每个标签所属组通知给标签。

它提出的协议包括：

- `TPG`：传统逐标签轮询，每个标签都发送标签 ID 和组 ID。
- `EPG`：按组轮询，减少重复发送组 ID。
- `FIG`：用 Bloom Filter 表示某个组，减少标签 ID 广播，但要处理假阳性。
- `CCG`：并发分组，尝试利用碰撞让多个标签同时获得组 ID。

从 C1G2 角度看，第一篇论文能帮助理解：

- 为什么逐标签单播或逐标签访问很慢；
- 为什么减少 ID 传输、减少无效槽和利用批量筛选很重要；
- 为什么“碰撞”既是传统 RFID 的问题，也可能成为协议设计可利用的信号。

但要注意，第一篇中某些机制偏理论协议设计，不一定能直接在不改硬件、不改底层协议的商品化 C1G2 系统上部署。第二篇论文的价值就在于严格使用 C1G2 可用命令。

## 14. 第二篇论文与 C1G2 的关系

`Time-efficient Range Detection in Commodity RFID Systems` 的目标是：在商品化 C1G2 系统中判断是否存在数据落入给定范围的标签。

它的协议设计依赖以下 C1G2 能力：

| C1G2 能力 | 论文中如何使用 |
|---|---|
| User memory 可存应用数据 | 把待检测数据 `di` 编码并存入标签 |
| Select 可按 `Pointer/Length/Mask` 匹配 | 用掩码筛出同值标签或范围候选标签 |
| Action 可把匹配/不匹配标签置 A/B | 用 A 表示活跃候选，用 B 表示静默 |
| Session inventoried flag 可维持状态 | 用 S2 维护临时筛选结果 |
| Query 可只盘点 A 或 B 标签 | 只对候选集合执行 inventory |
| inventory 回复可判断集合非空 | 有回复即存在目标标签 |

这个思路的本质是：把标签端的复杂判断限制在标准允许的“比特串比较”上，把复杂的范围拆解与优化放在读写器/后端完成。

## 15. 学习时最容易混淆的点

### 15.1 Select 不是 Query

`Select` 是状态修改命令，不产生标签回复；`Query` 才会让标签参与盘点并回复 RN16。

### 15.2 Query 不是按内存筛选

`Query` 只能根据 `SL`、session 和 inventoried flag 选择参与标签。按内存内容筛选必须先用 `Select` 完成。

### 15.3 Read 不是批量查询工具

`Read` 是访问单个已识别标签的命令。范围检测如果对每个标签都 Read，就会退化成低效逐标签读取。

### 15.4 Pointer 与 WordPtr 不同

`Select.Pointer` 是 bit 地址；`Read.WordPtr` 是 16-bit word 地址。论文中的掩码位置与标准中的内存地址换算时要小心。

### 15.5 A/B 不是永久标签属性

`A` 和 `B` 只是某个 session 中的临时 inventoried flag 状态。不同 session 独立，且 flag 有持久时间限制。

### 15.6 目标范围不是由标签计算大小关系

标签不会执行 `di <= τ` 这样的数值比较。读写器需要把范围条件拆成标签能执行的掩码匹配。

## 17. 一页总结

C1G2 对协议设计最重要的思想可以压缩成一句话：

> 标签很弱，不能执行复杂计算；读写器很强，负责把复杂任务拆成标签能执行的状态切换、掩码匹配和随机盘点。

对两篇论文来说：

- 第一篇展示了 RFID 协议设计如何围绕“减少逐标签通信、减少无效槽、处理碰撞”展开。
- 第二篇展示了如何在 C1G2 标准限制内，用 `Select + Query` 实现高效范围检测。

最需要掌握的标准机制是：

- 四个 memory bank，尤其是 User memory；
- session 与 inventoried flag 的 A/B 状态；
- SL flag 与 Query 的 Sel 字段；
- Select 的 `Target / Action / MemBank / Pointer / Length / Mask`；
- Query 的 `Session / Target / Q`；
- slot counter、RN16、ACK 与 singleton/collision/empty slot；
- Access 流程中的 `Req_RN / Read / Write`。

掌握这些内容后，再看 `Range Detection` 中的 SQ 和 RQ，就能理解作者不是发明了新硬件能力，而是在标准命令之间找到了更高效的组合方式。
