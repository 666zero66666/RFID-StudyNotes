# Fast RFID Grouping Protocols 全文翻译

原文信息：

- 标题：Fast RFID Grouping Protocols
- 作者：Jia Liu, Bin Xiao, Shigang Chen, Feng Zhu, Lijun Chen
- 会议：IEEE INFOCOM 2015

## 摘要

在 RFID 系统中，分组问题是指：给定一个划分方案，如何高效地把所有标签划分到相应组中，使同一组中的标签拥有相同的组 ID。与已有工作主要研究读写器到单个标签的单播传输不同，分组为大规模 RFID 应用中的高效组播传输和聚合查询提供了一种基础机制。在组播中，一条消息可以同时传给同一组中的 `m` 个标签，因此与单播相比，效率可以提升 `m` 倍。

本文研究大规模 RFID 系统中的快速分组协议。据作者所知，这是第一次系统性地处理这个在实践中重要、但此前尚未被深入研究的问题。文章首先给出一个直接方案，称为增强轮询分组协议（Enhanced Polling Grouping, EPG）。随后提出一个时间效率更高的过滤分组协议（Filtering Grouping, FIG），它利用 Bloom Filter 消除代价高昂的标签 ID 传输。作者进一步指出，基于 Bloom Filter 的方案由于其固有的假阳性问题存在局限，因此又提出最终方案并发分组协议（Concurrent Grouping, CCG）。CCG 采用了一种截然不同的设计：它通过利用碰撞，使多个标签能够同时获知自己的组 ID；同时通过去除帧执行中的无效时隙，使性能优于 FIG。仿真结果表明，相比基线轮询协议，性能最好的 CCG 可将执行时间降低到原来的约 `1/11`。

## 1. 引言

无线射频识别（RFID）已经被广泛部署在多种应用中，例如被标记物体的监测与跟踪、供应链管理以及仓库库存控制。对 RFID 标签进行分组，能够在很大程度上提升 RFID 应用的性能。

例如，当属于同一组的标签共享一个公共组 ID 时，读写器就可以把相同的数据同时发送给这一组标签，相比传统的单播方式，通信开销会大幅下降。再例如，在所有标签完成分组之后，读写器可以针对某个组执行有效的聚合查询，例如采集传感器数据或估计该组标签的基数，这将显著有利于监测功能和库存管理。

设 RFID 系统中的标签总体为 `G`，其上给定一个任意的组划分，也就是 `G` 被划分成若干两两不重叠的子集。所谓分组问题，就是如何高效地把“你属于哪个组”这一信息通知给 `G` 中的所有标签，使同一子集中的标签具有相同的组 ID。

为进一步说明这一问题及其实际意义，论文给出了一个组播场景。读写器希望把数据 `D1` 发给标签 `t1、t2、t3`，把数据 `D2` 发给标签 `t4、t5、t6`。数据可以是需要写入标签、用于后续追踪的发运信息，也可以是要求标签上报不同传感信息的查询。传统做法是逐个标签单播，因此总共需要 6 次数据发送。相比之下，如果这些标签已经被分组，读写器就可以通过一次带有组 ID 目的地址的发送，把 `D1` 发给 `t1、t2、t3`；同理，再用一次发送把 `D2` 发给 `t4、t5、t6`。这样总共只需要 2 次发送，开销降低到原来的 `1/3`。因此，标签分组是能够大幅提升 RFID 应用管理效率的基础机制。

如何高效地对 RFID 标签进行分组，在此前仍是一个新问题。一个直观解法是借助传统轮询协议：对于每个标签，读写器发送它的标签 ID 以及分配给它的组 ID，以此让该标签知道自己属于哪个组。然而这个协议在大型 RFID 系统中效率很低，因为读写器必须广播大量标签 ID，以及同样数量的组 ID。

因此，本文提出一系列协议来逐步改进分组性能。首先，提出增强轮询分组协议 EPG，它避免了对同一个组 ID 的重复发送，从而优于传统轮询分组协议。接着，提出过滤分组协议 FIG，它利用 Bloom Filter 来避免发送标签 ID。针对 Bloom Filter 固有的假阳性问题，作者进一步通过联合优化求出系统最优参数，以使执行时间最小化。最后，提出可扩展性和效率都更高的并发分组协议 CCG。CCG 不再采用 Bloom Filter，而是能在单个时间帧中同时给不同组的标签赋予各自的组 ID，这与 FIG 的“每次只处理一个组”的思路有本质区别。此外，CCG 还能够把碰撞利用起来，使多个标签在一个时隙内同时完成标记；并借助一个排序向量去除任何被浪费的时隙，从而进一步提升效率。

文中推导了 CCG 的执行时间上界，该上界相当于只需传输 `(0.028 + 0.018 × ceil(log2 k)) × n` 个标签 ID 的代价，明显快于传统轮询协议中必须发送 `n` 个标签 ID 再加 `n` 个组 ID 的方式。其中 `n` 是系统标签总数，`k` 是组数。

作者基于 EPCglobal Gen-2 标准进行了大量仿真。结果显示：在 100 个组、10000 个 RFID 标签的场景下，传统轮询协议需要 `44.6s`；EPG 把时间降到 `39.1s`；FIG 进一步缩短到 `7.4s`；CCG 性能最好，仅需 `3.9s`，相当于相对传统轮询协议提升了 11 倍，因此更适合实时 RFID 应用。

文章后续结构如下：第 2 节给出问题陈述和一个直接解法；第 3 节提出 FIG；第 4 节提出更高效的 CCG；第 5 节进行性能评估；第 6 节总结全文。

## 2. 问题陈述

### 2.1 系统模型

一个 RFID 系统一般由一个或多个读写器以及其覆盖范围内的大量标签组成。读写器连接到后端服务器，用于信息存储和计算。每个标签拥有唯一的标签 ID。标签可以直接与读写器通信，但标签之间不能彼此通信。

如果多个读写器之间能够良好地同步和调度，那么逻辑上可以把它们视为一个整体。为了简化描述，本文中的协议先以单个读写器为对象给出，但在建立读写器之间无碰撞传输调度的前提下，它们也可以很容易地修改为多读写器版本。

本文假设读写器事先知道所有标签 ID。这一前提可以通过现有的众多标签识别协议自动获得。

在大型 RFID 系统中，被标记物体可以根据其属性，例如鞋类、箱包，到达或离开的日期，或其他标准，被划分到不同组中。对物体分组便于库存处理，也有利于仓库管理，因为可以基于组 ID 方便地对某些组执行操作。例如，当标签已经知道自己的组 ID 之后，读写器就可以把一条消息通过组 ID 作为目的地址发送给某一组内的所有标签，而不是通过标签 ID 给组内每个标签分别发一条消息。

### 2.2 问题定义

考虑一个包含 `n` 个标签的大型 RFID 系统，标签集合记为：

```text
G = {t1, t2, ..., tn}
```

集合 `G` 的一个划分，是一组两两不相交的集合族：

```text
P = {P1, P2, ..., Pk}
```

并满足所有 `Pi` 的并集等于 `G`。文中把每个 `Pi` 称为一个组，而且 `G` 中每个标签恰好属于其中一个组。

所谓分组问题，就是根据划分 `P`，高效地给 `G` 中所有 RFID 标签打标，使同一组中的标签拥有相同的组 ID。更具体地说，用户把划分 `P` 提供给读写器，读写器需要给同一组 `Pi` 中的所有标签赋予同一个组 ID `gi`，其中 `1 ≤ i ≤ k`，而且不同组必须使用不同的组 ID。

在现实系统中，如果是在部署前逐个手工写标签，则可以把标签 ID 的某个前缀作为静态组 ID。本文研究的是另一类问题：标签部署之后，面对任意给定的划分进行动态分组。理论上，仍然可以把标签 ID 前缀中的一部分作为组 ID 区域，但这样每次重新分组时都必须改写这一段。现有工作并没有回答：当所有标签已经混在一起时，如何高效地把每个标签新的组 ID 通知给它。

### 2.3 一个朴素方案

如前所述，在传统轮询分组协议（Traditional Polling Grouping, TPG）中，读写器首先通过广播某个标签的 ID 把它从其他标签中区分出来，然后再发送相应的组 ID 给该标签。对于属于同一组的多个标签，同一个组 ID 会被反复广播多次。

基于这一观察，作者先提出增强轮询分组协议（Enhanced Polling Grouping, EPG），用于避免冗余的组 ID 发送。

EPG 包含 `k` 个分组轮次，其中 `k` 是组的个数。每一轮只处理一个组。考虑任意一轮，用于分组 `Pi`，其中 `1 ≤ i ≤ k`。读写器依次广播属于 `Pi` 的所有标签 ID。系统中每个尚未被标记的标签持续监听信道，只有当某个标签收到的 ID 与自己的 ID 匹配时，它才会从“未标记”状态转移到“已标记”状态。等 `Pi` 中的所有标签都被轮询过之后，读写器广播组 ID `gi`，使这些已经处于“已标记”状态的标签转移到“已完成标记”状态。此后，每个已经完成标记的标签保持静默，其他标签继续保持可参与状态，用于后续轮次。

TPG 与 EPG 的主要差异在于：EPG 对每个组 ID 只发送一次。设 `tid` 表示传输一个 96 位标签 ID 所需时隙长度，`tgid` 表示传输一个组 ID 所需时隙长度，则：

- TPG 的总执行时间为 `n × (tid + tgid)`；
- EPG 的总执行时间为 `n × tid + k × tgid`。

尽管 EPG 相比 TPG 已经有所提升，但它仍然必须艰难地发送 `n` 个标签 ID，因此在大型标签集合下运行时间仍然很长。于是，作者继续寻找更快的分组协议。

## 3. 过滤分组协议 FIG

这一节提出一个高效的过滤分组协议 FIG，它可以避免大部分 ID 广播。

### 3.1 基本思想

FIG 的思路是：一次把一个组与其他组分离开，并借助一种空间效率高的 Bloom Filter 来实现这一点。

当读写器把一个编码了某个组的过滤器广播给系统中的所有标签时，属于这个目标组的标签都会被正确标记。但由于 Bloom Filter 存在假阳性，其他组中的某些标签也可能被错误地标记。由于读写器同时掌握过滤器和全部标签 ID，它可以预测出哪些标签被误标记，因此可以在后续附加阶段通过发送这些标签的 ID，通知它们取消标记。不过，这一附加阶段可能引入显著开销。

要降低去误标记的开销，可以通过增大过滤器长度来降低假阳性率；但这样会增加过滤阶段本身的开销。因此关键在于联合优化过滤开销和去误标记开销，使两者之和最小。最终得到的协议将显著优于 EPG。除此以外，还必须考虑不同组应用 Bloom Filter 的顺序，因为它同样影响总执行时间。

### 3.2 协议概览

FIG 包含 `k` 个分组轮次，每一轮处理划分 `P` 中的一个组。每一轮由三个阶段组成：

1. 过滤阶段：读写器广播一个对某个组进行编码的过滤器，只有通过该过滤器的标签才会从“未标记”状态进入“已标记”状态。
2. 轮询阶段：这一阶段的任务是去除假阳性造成的误标记标签。读写器广播这些标签的 ID，收到自己 ID 的标签重新回到“未标记”状态。
3. 标记阶段：读写器广播该组的组 ID，剩下处于“已标记”状态的真正目标标签据此进入最终“已完成标记”状态。

之后，尚未标记的标签继续参与后续轮次。

### 3.3 协议细节

考虑第 `i` 轮，该轮要处理的组记为 `P'i`，其中 `1 ≤ i ≤ k`。之所以写成 `P'i` 而不是 `Pi`，是为了强调：协议的处理顺序不必与 `P` 中组的定义顺序一致。记 `Gi` 为这一轮开始时仍处于“未标记”状态的标签集合。

#### 1. 过滤阶段

这一阶段利用 Bloom Filter，快速但近似地把 `P'i` 中的所有标签标记出来。读写器首先对 `P'i` 中每个标签的 ID 做哈希，把它们编码到一个长度为 `Li` 位、使用 `ki` 个哈希函数的 Bloom Filter 中，记作 `BF(P'i)`。其中 `Li` 和 `ki` 的最优取值稍后给出。

读写器先广播 `Li` 与 `ki` 的取值，再广播过滤器 `BF(P'i)`。如果过滤器太长，则可以把它拆成若干个 96 位段，每段在一个长度为 `tid` 的时隙内发送。集合 `Gi` 中每个未标记标签根据自己的 ID 用相同的 `ki` 个哈希函数映射到过滤器中的若干位，并因此知道自己需要记录过滤器中的哪些片段。若这 `ki` 个位置在 `BF(P'i)` 中全部为 1，则该标签通过过滤器，进入“已标记”状态；否则保持“未标记”状态。把所有处于“已标记”状态的标签集合记为 `Mi`，显然有 `Mi ⊆ Gi`。

#### 2. 轮询阶段

Bloom Filter 不会产生假阴性，因此 `P'i ⊆ Mi`。但是它会产生假阳性，也就是说，`Mi` 中有些标签并不属于 `P'i`。由于读写器知道 `BF(P'i)` 和当前未标记集合 `Gi`，它可以预测出 `Mi - P'i` 这一批应该被取消标记的标签。

接着，读写器依次广播这些误标记标签的 ID。那些收到自己 ID 的错误标签会从“已标记”状态退回到“未标记”状态。这样，在轮询阶段结束后，仍然留在“已标记”状态中的标签就全部属于 `P'i`。

#### 3. 标记阶段

最后，读写器广播 `P'i` 的组 ID，以通知所有当前“已标记”的标签：它们属于哪个组。收到组 ID 之后，这些标签转入“已完成标记”状态。随后协议进入下一轮。

### 3.4 最优参数设置

作者在下面的定理中给出 `Li` 和 `ki` 的最优值。

**定理 1：** 设 `ni` 是集合 `Gi` 中尚未标记标签的数量，`m'i` 是当前目标组 `P'i` 中标签的数量。那么，对任意 `1 ≤ i ≤ k`，第 `i` 轮的最优过滤器长度 `Li` 以及最优哈希函数数目 `ki` 为：

```text
ki = ln2 × Li / m'i

Li = m'i / (ln2)^2 × ln(96 × (ln2)^2 × (ni - m'i) / m'i)
```

在这一最优设置下，第 `i` 轮执行时间最小，为：

```text
T(m'i, ni) = Li/96 × tid + (ni - m'i) × 0.6185^(Li/m'i) × tid + tgid
```

**证明思路翻译如下：**

在第 `i` 轮中，过滤阶段传输一个 `Li` 位过滤器，需要的时间是 `Li / 96 × tid`。在轮询阶段，需要轮询的误标记标签数量等于 `|Gi - P'i| × fi`，其中 `fi` 是 Bloom Filter 的假阳性率。由于 `P'i ⊆ Gi`，所以 `|Gi - P'i| = ni - m'i`。因此轮询阶段的时间开销是：

```text
(ni - m'i) × fi × tid
```

标记阶段需要一个时隙传输组 ID，时间是 `tgid`。因此，第 `i` 轮总执行时间为：

```text
T(m'i, ni) = Li/96 × tid + (ni - m'i) × fi × tid + tgid
```

在 `Li、ni、m'i` 固定时，总执行时间随假阳性率 `fi` 单调增加，因此首先应该尽量减小 `fi`。Bloom Filter 的假阳性率满足：

```text
fi = (1 - (1 - 1/Li)^(ki m'i))^ki
   ≈ (1 - e^(-ki m'i / Li))^ki
```

对 `fi` 求一阶导并令其为零，可以得到最优的 `ki = ln2 × Li / m'i`，同时最小假阳性率为：

```text
fi = 0.6185^(Li/m'i)
```

代回总执行时间式中后，再对 `Li` 求导并令导数为零，可得文中给出的最优 `Li`。于是就得到上面的最小执行时间公式。

文中还说明：为了清晰表达，广播 `Li` 和 `ki` 本身的开销被忽略，因为它们通常只需要几个字节即可编码。

### 3.5 分组顺序

尽管单轮的最优参数已经可以由上面的公式得到，不同组的执行顺序仍然会导致不同的全局执行时间。所谓一个“组序列”，就是这些组在多个轮次中被处理的先后顺序。

寻找最优顺序并不容易。直接办法是遍历 `P` 中各组的所有排列，分别计算执行时间，再取最优。但 `k` 个组共有 `k!` 个排列，因此这一做法不可扩展。

为此，作者提出一个贪心组排序方案，用于找到近似最优的组顺序。其思想是：在每一步都把“当前轮分组开销最小”的候选组作为下一个要处理的组。

更形式化地说，设贪心得到的有序分组序列为：

```text
P'1, P'2, ..., P'k
```

对应各组的规模为：

```text
m'1, m'2, ..., m'k
```

并满足：

```text
T(m'i, ni) ≤ T(m'j, ni), 1 ≤ i ≤ j ≤ k
```

其中 `T(m'j, ni)` 表示：在第 `i` 轮、当前剩余未标记标签为 `ni` 的情况下，若选择组 `P'j` 来处理，则其执行时间是多少。条件 `T(m'i, ni) ≤ T(m'j, ni)` 表明，在所有尚未处理的组中，`P'i` 是当前轮开销最小的选择。

考虑第 `i` 轮。此时还剩下 `k - i + 1` 个未处理组。设这些组的规模为：

```text
q1, q2, ..., qk-i+1
```

并假设它们满足：

```text
q1 ≤ q2 ≤ ... ≤ qk-i+1
```

接着有如下结论。

**定理 2：** 下一轮要处理的组，一定是规模最小的组 `q1` 或规模最大的组 `qk-i+1` 之一。

**证明翻译如下：**

把某个未处理组的规模记作实数 `x > 0`，于是该轮执行时间可以看成函数 `T(x, ni)`。令其一阶导数为 0，可得最大值/最小值出现的位置：

```text
x = 0.7369 × ni
```

文中通过 Lambert W 函数推导这一结果。然后比较可知：

- 当 `x → 0` 时，`T(x, ni) → tgid`
- 当 `x = 0.7369 × ni` 时，`T(x, ni)` 取得最大值

也就是说，函数 `T(x, ni)` 会随着 `x` 的增大先上升，在 `0.7369 × ni` 达到峰值，然后再单调下降。因此，在一串按升序排列的候选规模 `q1 ≤ q2 ≤ ... ≤ qk-i+1` 中，最小执行时间一定出现在两端，也就是 `T(q1, ni)` 或 `T(qk-i+1, ni)`。

基于这个性质，作者给出一个简单的贪心排序算法：

1. 初始化有序集合 `M' = ∅`
2. 在每一步，从剩余组规模集合 `M` 中找出最小值 `x-` 和最大值 `x+`
3. 计算它们在当前剩余标签总数 `ni = sum(M)` 下的执行时间
4. 如果 `T(x-, ni) ≤ T(x+, ni)`，则选 `x-`；否则选 `x+`
5. 把被选中的组移出 `M`，加入 `M'`
6. 重复直到所有组都被排完

文中后面的实验会说明：这个贪心顺序非常接近真正的最优顺序。

## 4. 并发分组协议 CCG

### 4.1 动机

虽然 FIG 已经通过 Bloom Filter 显著提升了分组性能，但它仍然必须一次只处理一个组。一个过滤器确实能成功标记它编码的目标组 `P'i` 中的所有标签，但由于假阳性，那些不属于 `P'i` 的标签也可能被误标记。从逻辑上说，这可以视为过滤器内部发生了“碰撞”：目标组外部的某些标签与目标组中的标签在过滤器中映射到了相同模式。

问题在于：在很多场景中，目标组外部的标签数量远大于目标组内部的标签数量，这意味着误标记的非目标标签数甚至可能比目标组标签数还多，从而导致很高的轮询去误标记开销。除非把 Bloom Filter 的假阳性率压得非常低，但那又会带来更长的过滤器。

作者指出，这一两难本质上同时来自两个设计选择：

- 使用 Bloom Filter
- 采用“一次处理一个组”的策略

为了进一步提升性能，作者尝试探索完全不同的思路，而且不再使用 Bloom Filter。这些新思路包括：

- 把所有组的标签一起处理
- 使某些碰撞变成有用的碰撞，从而在一个时隙内给多个标签打标
- 提前识别那些无法利用的碰撞，并在真正执行时避免它们产生开销

这些想法共同构成了下一种协议，也就是并发分组协议 CCG。

### 4.2 协议描述

CCG 同样由多个分组轮次组成。每一轮包含两个阶段：

- 排序阶段（ordering phase）
- 标记阶段（labeling phase）

排序阶段告诉标签：自己是否会在当前轮被标记，以及如果会，会在什么时候被标记。标记阶段则向相应标签发送组 ID。

#### 1. 排序阶段

读写器首先广播一个带参数 `<f, r>` 的请求，其中：

- `f` 是虚拟帧中的时隙数
- `r` 是随机种子

这里要特别注意：这个虚拟帧不会真的被逐个时隙执行。它只是一个逻辑工具，用来找出哪些时隙是“有用的”，然后只真正执行这些有用时隙组成的实际帧。

收到该请求后，每个尚未标记的标签根据下面的规则随机选择一个时隙：

```text
slot = H(id, r) mod f
```

其中 `id` 是标签 ID，`H(·)` 是哈希函数。

如果某个时隙没有被任何标签选中，就叫空槽（empty slot）；如果恰好只有一个标签选中，就叫单标签槽（singleton slot）；如果有多个标签选中，就叫碰撞槽（collision slot）。

对 CCG 来说，“有用的时隙”是这样的时隙：

- 要么是单标签槽
- 要么是碰撞槽，但其中所有标签都来自同一个组

这类时隙被称为同构槽（homogeneous slot）。相对地，如果某个碰撞槽中的多个标签来自不同组，那么它就是异构槽（heterogeneous slot），这种时隙没有用。空槽显然也没有用。

举例来说，如果第一个槽由 `t1` 和 `t4` 选中，而它们恰好属于同一组 `P1`，那么第一个槽就是同构槽；若第三个槽由 `t2` 和 `t5` 选中，而它们属于不同组，则第三个槽就是异构槽。

选择到同构槽的标签称为同构标签（homogeneous tags）。每个标签自己并不知道自己选中的槽是不是同构槽，但读写器知道。因为读写器掌握所有标签 ID 和组信息，所以它能够预测虚拟帧中每个时隙是空槽、同构槽还是异构槽。

在真正执行标记帧之前，读写器会先删除所有空槽和异构槽，只保留同构槽。为了让标签知道哪些槽被保留，读写器广播一个长度为 `f` 位的排序向量 `V`，虚拟帧中的每个槽对应一位：

- 若该槽是空槽或异构槽，则相应位为 `0`
- 若该槽是同构槽，则相应位为 `1`

如果 `V` 太长，也可以拆成若干个 96 位片段，并以 `tid` 为单位逐段传输。

从标签视角看，排序向量 `V` 传递两类信息：

1. 标签通过查看自己所选时隙对应的那一位，知道自己选中的槽是不是同构槽。只有当这一位是 1 时，标签才从“未标记”状态进入“已标记”状态。
2. 排序向量还隐含了：自己对应的同构槽在后续真实标记帧中是第几个有效槽。若某个标记标签发现自己的那一位之前共有 `i` 个 1，那么它就知道自己对应真实帧中的第 `i+1` 个同构槽。

#### 2. 标记阶段

只有处于“已标记”状态的标签参与这一阶段。设同构槽总数为 `h`。读写器只对这 `h` 个同构槽发起一个实际的标记帧，并在每个槽中发送一个组 ID。

由于同构槽中所有标签都属于同一个组，因此读写器可以在一个时隙内同时给这些标签发送同一个组 ID。根据排序向量，每个已标记标签都知道自己应当监听真实帧中的哪个槽，从而从该槽中接收自己的组 ID。

这样，CCG 便实现了：

- 多个组一起参与
- 某些碰撞被有效利用
- 真正执行时没有空槽或无用碰撞槽

在标记阶段结束后，当前轮完成，随后继续下一轮，直到所有标签都完成标记。

### 4.3 参数设置与性能分析

现在需要决定虚拟帧大小 `f` 的最优取值。考虑任意一轮分组。记：

- `h` 为同构槽数量
- `η` 为同构标签数量，也就是落在这些同构槽中的标签总数

那么本轮执行时间为：

```text
t = f/96 × tid + h × tgid
```

作者定义分组效率 `λ` 为：单位执行时间内成功被处理的同构标签数：

```text
λ = η / t = η / (f/96 × tid + h × tgid)
```

显然，`λ` 越大，表示单位时间内能完成标记的标签越多。因此问题变成：找出使 `λ` 最大的 `f`。

设本轮开始时，各组中尚未标记的标签数分别为：

```text
m'1, m'2, ..., m'k
```

它们的总和记作：

```text
n' = Σ m'i
```

接下来作者先分别推导同构槽数量和同构标签数量的期望。

#### 定理 3：同构槽数量期望

若虚拟帧有 `f` 个时隙，则同构槽期望数量为：

```text
h = f × Σ[(1 - 1/f)^(n' - m'i) × (1 - (1 - 1/f)^(m'i))]
```

**证明含义：**

对某个组 `Pi` 来说，某个时隙要成为“只被 `Pi` 中标签占据的同构槽”，必须满足两件事：

1. `Pi` 中至少有一个标签选到这个槽
2. 所有不属于 `Pi` 的标签都不能选到这个槽

前者的概率是：

```text
1 - (1 - 1/f)^(m'i)
```

后者的概率是：

```text
(1 - 1/f)^(n' - m'i)
```

两者相乘就是这个槽仅由组 `Pi` 的标签选中的概率。再对 `k` 个组求和，并乘以总槽数 `f`，就得到期望同构槽数量。

#### 定理 4：同构标签数量期望

在虚拟帧大小为 `f` 时，同构标签的期望数量为：

```text
η = Σ[m'i × (1 - 1/f)^(n' - m'i)]
```

**证明含义：**

看某一组 `Pi`。如果某个时隙中恰好有 `Pi` 的 `j` 个标签选到它，而所有其他组标签都没选到它，那么这 `j` 个标签都会成为同构标签。对 `j = 0 ... m'i` 逐一求期望后，可以得到每个时隙中来自 `Pi` 的同构标签期望值，再乘以 `f`，最终化简为上式。

把 `h` 与 `η` 代回分组效率 `λ` 的定义，就可以得到 `λ` 关于 `f` 的表达式。虽然很难直接从闭式表达式中解析求得最优 `f`，但作者进一步给出了一个足够用的最优区间。

#### 定理 5：最优 `f` 的范围

当 `λ` 取得最大值时，`f` 一定落在区间：

```text
[1, e(n' + 1)]
```

其中 `e` 是自然常数。

**证明思路翻译如下：**

作者比较两个不同帧大小 `f1 ≤ f2` 下的分组效率 `λ(f1)` 与 `λ(f2)`。通过对：

- `η1、η2` 的上下界
- `h1、h2` 的上下界

进行估计，证明当 `f2 ≥ e(n' + 1)` 时，一定可以找到一个 `f1 = n' + 1` 使得 `λ(f1) ≥ λ(f2)`。这意味着：任何大于 `e(n' + 1)` 的 `f` 都不可能优于某个较小取值，因此真正的最优值一定落在 `[1, e(n' + 1)]` 中。

基于这个结论，系统就可以在该区间内用数值方法搜索使 `λ` 最大的 `f`。

#### 定理 6：CCG 执行时间上界

作者无法给出总执行时间的闭式精确表达式，但给出了一个上界：

```text
Tupper = n × (e/96 × tid + (e - 1) × tgid)
```

约化后可写作：

```text
Tupper ≈ n × (0.028 × tid + 1.718 × tgid)
```

**证明思路翻译如下：**

根据定理 5，某个最优 `f` 至少不劣于 `f1 = n' + 1`。作者进一步估计：

- 单标签槽必定是同构槽，因此同构标签数有一个下界
- 非空槽数量给出了同构槽数的上界

由此得到 `λ(f1)` 的一个下界，于是总执行时间 `n / λ(f1)` 给出上界。最终得到上述公式。

### 4.4 性能进一步改进

在标记阶段，读写器需要在每个同构槽中广播一个组 ID。当组 ID 较长时，这一步会变得耗时。为此，作者提出一个轻量级改动，使标记阶段对组 ID 长度不那么敏感。

关键思想是：不直接发送组 ID `gi`，而发送组的索引 `i`。如果系统总共有 `k` 个组，那么只需要 `ceil(log2 k)` 位即可区分所有组。例如，当 `k = 2` 时，每个组仅需 1 位索引。

进一步地，读写器可以把所有同构槽对应的组索引按顺序串接起来，形成一个长度为 `ceil(log2 k) × h` 的标记向量，其中 `h` 是同构槽数量。若该向量过长，也可以按 96 位切分逐段传输。第 `i` 个同构槽中的标签将从该向量中的第 `i` 段索引位置上获取自己的组索引。

这样一来，前面公式中的 `tgid` 就可以替换为：

```text
ceil(log2 k) / 96 × tid
```

从而得到 CCG 总执行时间更紧的上界：

```text
(0.028 + 0.018 × ceil(log2 k)) × tid × n
```

## 5. 性能评估

这一节中，作者通过仿真评估 EPG、FIG 与 CCG 的性能。先验证：

- FIG 中贪心组排序方案的有效性
- FIG 与 CCG 执行时间推导的准确性

然后把三种协议与基线协议 TPG 做对比。

### 5.1 仿真设置

仿真中，系统共有 `k` 个组，每个组的标签数量按照正态分布 `N(μ, σ)` 随机生成，其中 `μ` 为均值，`σ` 为标准差。

通信参数遵循 EPCglobal Gen-2 标准。任意两次连续通信，无论是读写器到标签还是标签到读写器，中间都要间隔 `302μs`。读写器到标签的数据率范围为 `26.7 kbps` 到 `128 kbps`。作者采用下界 `26.7 kbps`，并指出在其他参数设置下也能得到相似结论。按这一速率计算，读写器发送 1 比特需要 `37.45μs`，因此发送一个 96 位标签 ID 的时长为：

```text
tid = 37.45 × 96 + 302 = 3897.2μs
```

仿真中，组 ID `gi` 被设为组索引 `i`，其长度为 `ceil(log2 k)` 位，因此：

```text
tgid = 37.45 × ceil(log2 k) + 302 μs
```

文中的结果均取 200 次独立试验的平均值。

### 5.2 协议验证

#### 1. 贪心组排序方案

第 3 节已讨论：FIG 的全局执行时间会受到组处理顺序影响。作者在图 7(a) 中比较了四种组顺序：

- 最优顺序
- 贪心顺序
- 随机顺序
- 最差顺序

实验设置为 4 个场景：

- 场景 1：`k = 5`，组规模分布 `N(1000, 800)`
- 场景 2：`k = 5`，组规模分布 `N(1000, 100)`
- 场景 3：`k = 8`，组规模分布 `N(1000, 800)`
- 场景 4：`k = 8`，组规模分布 `N(1000, 100)`

其中，随机顺序直接由随机生成的组序列给出；贪心顺序则以该随机顺序为输入，再通过算法 1 生成；最优顺序和最差顺序则通过穷举全部 `k!` 种可能性找出。

结果表明：

- 最优顺序最好
- 贪心顺序非常接近最优
- 随机顺序更差
- 最差顺序最慢

尤其是贪心顺序与最优顺序之间差距极小，说明算法 1 能够找到近似最优的组顺序。作者还指出，当不同组规模本来就比较接近时，执行时间对组顺序不太敏感。

#### 2. 执行时间公式验证

接着，作者分别对 FIG 和 CCG 的理论执行时间公式进行仿真验证。

测试包括 3 类场景：

- 场景 A：`k = 20, μ = 200, σ = 20`
- 场景 B：`k = 20, μ = 1000, σ = 100`
- 场景 C：`k = 100, μ = 200, σ = 20`

每个场景采样 200 次仿真结果，并计算相对误差：

```text
(vsim - vtheo) / vtheo
```

其中 `vsim` 为仿真时间，`vtheo` 为理论时间。

结果显示：

- 对 FIG，相对误差大多落在 `[-0.05, 0.05]`
- 随着组规模或组数增加，相对误差进一步减小
- 对 CCG，相对误差大多落在更小的 `[-0.02, 0.02]`

这说明推导出的理论时间能够较准确描述 FIG 与 CCG 的实际执行时间。

### 5.3 协议性能

作者进一步比较 EPG、FIG、CCG 与基线协议 TPG 在不同参数下的实际执行时间。

#### 1. 整体对比

图 8(a) 使用 3 个场景：

- 场景 1：`k = 50, μ = 100, σ = 40`
- 场景 2：`k = 100, μ = 100, σ = 40`
- 场景 3：`k = 100, μ = 200, σ = 80`

以场景 2 为例，此时总标签数 `n = 10000`。结果如下：

- TPG：`44.6s`
- EPG：`39.1s`
- FIG：`7.4s`
- CCG：`3.9s`

也就是说，在这一场景下：

- EPG 只是去掉了重复组 ID 广播，因此改进有限
- FIG 大幅压缩了标签 ID 广播开销
- CCG 又进一步把 FIG 的时间几乎减半，并把总时间压缩到 TPG 的约 `1/11`

在另外两个场景中也得到一致结论：CCG 最好，FIG 次之，EPG 再次，TPG 最差。

#### 2. 组数 `k` 的影响

图 8(b) 研究组数 `k` 对执行时间的影响。设置：

- 总标签数 `n = 2^14 = 16384`
- `σ = 0`
- `μ = n / k`

并让 `k` 从 2 增加到 1024。

结果显示：

- EPG 几乎保持稳定，因为发送组 ID 的开销相对 `n` 个标签 ID 的广播可忽略
- FIG 和 CCG 都随 `k` 呈对数增长

其中：

- FIG 的增长是因为组数变多，会导致更多分组轮次
- CCG 的增长是因为标记阶段需要发送 `ceil(log2 k)` 位的组索引

尽管二者都随着 `k` 增长而变慢，但 CCG 增长更缓慢，说明它对组数不如 FIG 敏感。

#### 3. 平均组规模 `μ` 的影响

图 8(c) 研究平均组规模 `μ` 的影响。设置：

- `k = 100`
- `σ = 0`
- `μ` 从 10 变化到 100

结果表明：

- EPG 随 `μ` 明显升高，当 `μ = 100` 时接近 `40s`
- CCG 在所有 `μ` 下都保持最小执行时间，不超过 `4s`
- FIG 虽然远优于 EPG，但仍慢于 CCG

因此可以得出：`μ` 越大，总标签数 `n = k × μ` 也越大，三种协议时间都会上升，但 CCG 对这一变化最不敏感。

#### 4. 标准差 `σ` 的影响

图 8(d) 研究标准差 `σ` 的影响。设置：

- `k = 10`
- `μ = 1000`
- `σ` 从 100 变化到 800

结果显示：

- EPG 和 CCG 基本稳定，因为总标签数和组数不变
- FIG 随 `σ` 增大略有下降

原因是：`σ` 越小，不同组规模越相似，读写器越难把属于同一组的标签快速筛出来；当 `σ` 增大时，组之间差异变大，FIG 略有受益。不过随着 `k` 增大，不同组规模间的相对差异会缩小，`σ` 的影响也会减弱。

综合图 8 中的结果，作者认为：

- 平均组规模 `μ` 或总标签数 `n` 对执行时间影响最大
- 其次是组数 `k`
- 标准差 `σ` 主要影响 FIG，对 EPG 和 CCG 基本没有影响

## 6. 结论

本文研究了 RFID 系统中“如何快速给大量标签分组”这一新问题。标签分组对于提升 RFID 应用中的库存与管理效率具有基础意义。作者针对这一问题提出了三种协议：

- EPG：相对传统轮询分组协议，避免反复发送相同组 ID
- FIG：用 Bloom Filter 代替繁琐的轮询，按组给标签打标
- CCG：避免 Bloom Filter 的假阳性缺陷，并在单个时间帧中同时给不同组标签赋值

仿真结果表明，CCG 的性能最好：它大约只需 FIG 一半的时间、EPG 的十分之一，以及基线协议 TPG 的十一分之一。
